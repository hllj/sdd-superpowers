# Feature 014: Tiered Memory Architecture

**Status:** Approved
**Created:** 2026-06-24
**Branch:** `014-tiered-memory-architecture`

---

## Problem Statement

Memory in SDD Superpowers is currently scattered across three always-loaded artifacts with overlapping concerns: `CLAUDE.md` carries meta-instructions alongside implicit project identity; `memory/constitution.md` duplicates that identity purpose; and `memory/steering/*.md` overlaps with `CLAUDE.md` on tech stack and conventions, leaving no single source of truth for any operational fact. The four reactive memory types (user/feedback/project/reference) blur in practice — nothing enforces dedup before a new entry is written — causing the MEMORY.md index to accumulate redundant entries across sessions.

## Goals

- Replace the flat memory layout with a three-tier model where each artifact has exactly one job and loading mechanism
- Rename `memory/constitution.md` → `memory/foundation.md` (Tier 0) and update all references
- Trim `CLAUDE.md` to a boot layer: skill invocation rule + memory pointer + hard gates only
- Make `sdd-init` generate both the CLAUDE.md boot layer and `memory/foundation.md` from a template + 4-question ceremony
- Establish enforceable dedup and conflict-resolution rules for Tier 2 reactive memory
- Migrate the sdd-superpowers repo itself (CLAUDE.md + constitution.md) in this feature

## Non-Goals

- Automated migration tool for user projects that already have `constitution.md` (blocked with a warning; manual rename only)
- Lint hook to enforce that CLAUDE.md does not repeat Tier 1 content (documentation-only convention)
- Semantic search or tag-based filtering across memory tiers
- Versioning or history tracking for memory entries
- `supersedes:` frontmatter field for Tier 2 entries
- Modifying the Tier 1 steering file schema or `loaded-by` convention

## Users and Context

**Primary users:** Developers using sdd-superpowers in a new project — running `sdd-init` for the first time
**Secondary users:** Claude itself — reading and writing memory entries across sessions
**Usage context:** Session start (loading Tier 0 + Tier 2 index), skill invocation (loading Tier 1), and any point in a session where a new fact is established (writing Tier 2)
**User mental model:** Memory should work like a filing system with clearly labeled drawers — one drawer for project identity (always open), one for tool-specific context (open when needed), one for session learning (indexed, on demand). No drawer should contain content that belongs in another.

## User Stories

### Story 1: Foundation File Generated by sdd-init

**As a** developer starting a new SDD project
**I want** `sdd-init` to run existence checks and generate `memory/foundation.md` from the 4-question ceremony
**So that** project identity lives in a single Tier 0 file that is always loaded — not split between CLAUDE.md and a separate constitution

**Acceptance criteria:**

- [ ] **AC-1.1** Given `sdd-init` is invoked and neither `memory/foundation.md` nor `memory/constitution.md` exists When the 4-question ceremony completes Then `memory/foundation.md` is written containing sections: `## Mission`, `## Principles`, `## Operational Context` (pointer to `memory/steering/`), and `## Amendment Process`
- [ ] **AC-1.2** Given `sdd-init` is invoked and `memory/foundation.md` already exists When the existence check runs Then `sdd-init` prints "foundation.md already exists — project already initialized" and exits without writing any files
- [ ] **AC-1.3** Given `sdd-init` is invoked and `memory/constitution.md` exists When the existence check runs Then `sdd-init` prints "old constitution detected — run migration before re-initializing" and exits without writing any files
- [ ] **AC-1.4** Given `sdd-init` is invoked and both `memory/constitution.md` and `memory/foundation.md` exist When the existence check runs Then `sdd-init` prints a conflicting-state warning and exits without writing any files
- [ ] **AC-1.5** Given `memory/foundation.md` has been written by `sdd-init` When a new Claude Code session starts Then the SessionStart hook loads `memory/foundation.md` into context (verifiable: hook configuration references `foundation.md`, not `constitution.md`)

### Story 2: CLAUDE.md Generated as Boot Layer

**As a** developer returning to a project
**I want** `CLAUDE.md` to contain only the minimum required to start working — skill invocation rule, memory pointer, hard gates
**So that** the always-loaded context is short and I do not need to scroll through a workflow diagram or skills table to understand how to begin

**Acceptance criteria:**

- [ ] **AC-2.1** Given `sdd-init` is invoked and `CLAUDE.md` does not exist When generation completes Then `CLAUDE.md` is written from `skills/sdd-init/templates/claude-md.md` and contains: the `<!-- sdd-init: generated -->` sentinel, the `sdd-superpowers:sdd-workflow` invocation rule, the memory pointer (`memory/MEMORY.md` index + `memory/foundation.md` identity), and the four hard gates
- [ ] **AC-2.2** Given `sdd-init` writes `CLAUDE.md` When the file is read Then `<!-- sdd-init: generated -->` appears as the first line of the file
- [ ] **AC-2.3** Given `CLAUDE.md` exists and its first line is `<!-- sdd-init: generated -->` When `sdd-init` runs the CLAUDE.md generation step Then `sdd-init` skips writing CLAUDE.md with no warning
- [ ] **AC-2.4** Given `CLAUDE.md` exists and does NOT contain `<!-- sdd-init: generated -->` When `sdd-init` runs the CLAUDE.md generation step Then `sdd-init` appends a `## Project Foundation` section to the existing file without overwriting it
- [ ] **AC-2.5** Given the generated `CLAUDE.md` template When it is read Then it does NOT contain any of: a skills table, a workflow diagram, tech stack descriptions, or mission/principles content

### Story 3: sdd-superpowers Repo Self-Migration

**As a** developer working on the sdd-superpowers plugin itself
**I want** the repo's own `memory/constitution.md` renamed to `memory/foundation.md` and `CLAUDE.md` trimmed to the boot layer
**So that** the plugin eats its own cooking — the repo demonstrates the tiered architecture it enforces

**Acceptance criteria:**

- [ ] **AC-3.1** Given `memory/constitution.md` exists in this repo When the migration is applied Then `memory/foundation.md` exists with the same Mission, Principles, Operational Context, and Amendment Process content, and `memory/constitution.md` is removed
- [ ] **AC-3.2** Given the migration is applied When `.claude/settings.json` is read Then the SessionStart hook references `memory/foundation.md` (not `memory/constitution.md`)
- [ ] **AC-3.3** Given the migration is applied When `CLAUDE.md` is read Then it contains only the boot layer content (sdd-workflow invocation rule + memory pointer + hard gates) and does NOT contain a skills table, workflow diagram, or tech stack section
- [ ] **AC-3.5** Given the migration is applied When `docs/contributing.md` is read Then it contains the skills table and workflow diagram that were removed from `CLAUDE.md`
- [ ] **AC-3.6** Given the migration is applied When `skills/sdd-init/SKILL.md` is read Then it references `foundation.md` and `foundation file` — not `constitution.md` or `constitution`
- [ ] **AC-3.7** Given the migration is applied When `memory/project_constitution_format.md` is read Then it references `foundation.md` and does not contain guidance to ignore the hook warning
- [ ] **AC-3.4** Given the PostToolUse:Write hook validates memory files When `memory/foundation.md`, `memory/MEMORY.md`, or any `memory/steering/*.md` file is written Then the hook does not report a frontmatter-missing error for those files

### Story 4: Tier 2 Dedup and Pruning

**As** Claude writing a memory entry during a session
**I want** clear rules for when to update an existing entry versus create a new one, and for when to prune the MEMORY.md index
**So that** the index stays under the 200-line limit and does not accumulate duplicate or near-duplicate entries

**Acceptance criteria:**

- [ ] **AC-4.1** Given Claude is about to write a new Tier 2 memory entry When `MEMORY.md` already contains an entry of the same type (user/feedback/project/reference) whose named entity overlaps with the new fact Then Claude updates the existing file and its MEMORY.md pointer line — it does not create a new file
- [ ] **AC-4.2** Given Claude is creating a genuinely new Tier 2 fact When the topic and named entity do not match any existing MEMORY.md entry Then Claude creates a new file and appends a new pointer line to MEMORY.md
- [ ] **AC-4.3** Given a new entry is appended to MEMORY.md When the total line count would exceed 200 Then the oldest pointer line (the topmost entry in the file) is removed from MEMORY.md while its backing `.md` file is preserved on disk

## Functional Requirements

### FR-1: Three-Tier Memory Model

The memory system must assign every artifact to exactly one tier with a single defined loading mechanism.

**Must:**
- Tier 0: `memory/foundation.md` — loaded by SessionStart hook at every session start
- Tier 1: `memory/steering/*.md` — loaded by individual skills via `loaded-by` frontmatter at skill invocation time
- Tier 2: `memory/*.md` + `memory/MEMORY.md` — MEMORY.md index loaded by SessionStart hook; individual files read on demand

**Must not:**
- Duplicate any fact across tiers (e.g., tech stack must not appear in both CLAUDE.md and a steering file)
- Introduce a new hook type — existing SessionStart hook is extended, not replaced

### FR-2: Foundation File Schema

`memory/foundation.md` must follow a defined schema so the SessionStart hook and skills can rely on consistent structure.

**Must:**
- Contain exactly these top-level sections in order: `## Mission`, `## Principles`, `## Operational Context`, `## Amendment Process`
- Begin with the line `> Loaded every session. To amend, follow the Amendment Process below.`
- NOT include YAML frontmatter (it is a project foundation file, not a memory entry)

**Must not:**
- Be created by any path other than `sdd-init` (or manual migration for this repo's own migration step)

### FR-3: CLAUDE.md Template

The CLAUDE.md template must live at `skills/sdd-init/templates/claude-md.md` and produce a minimal boot-layer file.

**Must:**
- Include `<!-- sdd-init: generated -->` as the first line
- Include the `sdd-superpowers:sdd-workflow` invocation instruction
- Include a pointer to `memory/MEMORY.md` (index) and `memory/foundation.md` (identity)
- Include the four hard gates verbatim

**Must not:**
- Include a skills table, workflow diagram, tech stack section, team practices, or mission/principles content

### FR-4: sdd-init Existence Checks

`sdd-init` must validate pre-conditions before writing any file.

**Must:**
- Check for `memory/foundation.md`, `memory/constitution.md`, and both simultaneously before any generation step
- Exit with a specific, human-readable warning message for each non-clean state (see Error Scenarios)
- Run existence checks as the very first action — before any user-facing questions

**Must not:**
- Write any file if any existence check fails
- Silently overwrite existing foundation.md or constitution.md

### FR-5: Contributing Guide

The sdd-superpowers repo must have a `docs/contributing.md` that captures plugin-contributor context removed from `CLAUDE.md`.

**Must:**
- Contain the skills table (skill name → when to use)
- Contain the workflow diagram (brainstorm → specify → plan → tasks → execute → review)
- Be created as part of the sdd-superpowers self-migration step in this feature

**Must not:**
- Be loaded automatically at session start — it is reference documentation, not operational context

### FR-6: Hook Whitelist for Foundation and Index Files

The PostToolUse:Write hook that validates memory entries must exempt non-entry files.

**Must:**
- Whitelist `memory/foundation.md`, `memory/MEMORY.md`, and `memory/steering/*.md` from frontmatter validation
- Still validate all other `memory/*.md` files for required frontmatter (name, description, type)

### FR-7: Tier 2 Dedup Rule

When writing Tier 2 memory entries, dedup must be checked before creating any new file.

**Must:**
- Scan MEMORY.md index before writing
- Apply the rule: same memory type AND overlapping named entity → update existing file; otherwise → create new file
- Prune oldest MEMORY.md pointer line (not backing file) when line count would exceed 200

## Non-Functional Requirements

### Performance

- Existence checks in `sdd-init` complete before any user-facing output — no observable delay
- SessionStart hook loads foundation.md and MEMORY.md atomically at session start — no deferred loading

### Security

- No credentials, tokens, or sensitive data in any memory tier — same policy as current memory system
- foundation.md amendment process requires explicit user approval — no automated overwrites

### Reliability

- `sdd-init` existence checks must run before any file write — no partial initialization states
- MEMORY.md pruning must remove the pointer line only — backing file must always be preserved
- sdd-superpowers repo migration must result in exactly one of `constitution.md` / `foundation.md` existing — never both, never neither

## Error Scenarios

| Scenario | Expected Behavior |
|----------|-------------------|
| `memory/foundation.md` exists when `sdd-init` runs | Print "foundation.md already exists — project already initialized" and exit; no files written |
| `memory/constitution.md` exists when `sdd-init` runs | Print "old constitution detected — run migration before re-initializing" and exit; no files written |
| Both `constitution.md` and `foundation.md` exist | Print conflicting-state warning naming both files; exit; no files written |
| `CLAUDE.md` has `<!-- sdd-init: generated -->` sentinel | Skip CLAUDE.md generation silently; continue with other artifacts |
| `CLAUDE.md` exists without sentinel | Append `## Project Foundation` section; do not overwrite existing content |
| MEMORY.md reaches 200 lines on new entry append | Remove oldest pointer line from MEMORY.md; preserve backing `.md` file on disk |
| PostToolUse:Write fires on `memory/foundation.md` | Hook recognizes whitelist match; no frontmatter-missing error emitted |
| PostToolUse:Write fires on a non-whitelisted `memory/*.md` without frontmatter | Hook reports error as before — no change to existing validation behavior |
